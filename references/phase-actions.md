# S3. 액션리스트 — 캐스케이드 도출

| 항목 | 내용 |
|------|------|
| 성격 | 마일스톤 달성에 필요한 구체적 아이템. 캐스케이드로 자동 도출 |
| 입력 | 로드맵 마일스톤 + 현재 상태 갭 분석 + 성장 프레임 + 도메인 전문지식 |
| 범위 | 제품 기능만이 아님. 비즈니스 액션 전체 포함 |
| 단위 | 주제별 독립 문서. 1문서 = 1트랙 |

## 캐스케이드 도출 절차

CEO가 아이템을 나열하지 않아도 빈칸을 채우는 것이 캐스케이드 목적. 반드시 이 순서:

①달성조건 나열 → ②현재 vs 달성조건 갭 식별 → ③성장 프레임 6단계 매핑 → ④도메인 전문지식으로 갭 해소 방법 도출 → ⑤시간순+의존성 정리 → ⑥파트 자동 분류(→파트 자동 분류 섹션)

## 서론 논리 설계

액션리스트는 "왜 이 아이템들이 필요한가"를 먼저 설명한다. 서론 = 독자 설득.

**성장 프레임(MECE 6단계):** ①만든다(Build) ②오게한다(Acquire) ③남게한다(Retain) ④퍼뜨리게(Refer) ⑤번다(Monetize) ⑥잰다(Measure). 각 액션리스트가 어느 단계를 담당하는지 서론에 명시. 빈 단계=갭=새 액션리스트 후보.

**서론 구성:** ①성장 프레임 전체 조감 → ②이 문서가 메우는 갭 → ③핵심 전략/원칙 → ④(선택) 예산/시뮬레이션

## 트랙 네이밍

**제목:** `액션리스트_[트랙명]_[시점]_v{M}.{m}`

트랙명 = 핵심 키워드 1개. 복수 액션리스트의 제목만으로 스토리라인 형성.

**아이템 번호:** 트랙 접두어 + 시간순 번호. `[접두어]-[번호]` (PA-1, G-5, U-3). 범용 번호(D1/D2) 금지.

## 아이템 테이블

전체 아이템을 테이블로 조감. 시간순 배열이 기본. 의존성 있으면 테이블 위에 트리 배치.

| # | 액션 아이템명 | 파트 | 엔진/카테고리 | 등급 | 상태 |

**파트 열:** 기획/디자인/개발 (파트 자동 분류에서 자동 분류된 결과)
**등급:** S(핵심/선행), A(표준), B(후순위)
**상태:** 미착수 → 방향메모 → 브리프작업중 → 브리프완료 → 팀전달

## 아이템 카드

필수=한줄정의+Why+So What. 옵시디언 기본→콜아웃 카드(`> [!info]`), 애플디자인 모드→HTML div 카드(apple-design-style), div 스타일 지정시→HTML div 카드(html-div-style). 대화→마크다운 표.

## 크로스맵

복수 액션리스트시 트랙 간 연결 표시(단방향 `→`). 템플릿 → `references/html-templates.md`.

## 파트 자동 분류

모든 액션 아이템은 실행 성격에 따라 3파트 중 하나로 자동 분류된다. 팀에 넘길 때 담당 파트가 명확해야 실행 병목이 줄어들기 때문이다.

| 파트 | 판단 기준 | 예시 |
|------|----------|------|
| 기획 | 사용자 시나리오, 정책, 요구사항 정의, 스펙 결정 | 프로필 항목 정의, 매칭 규칙 설계, 정산 정책 |
| 디자인 | UI/UX, 화면 설계, 인터랙션, 비주얼 | 프로필 입력 화면, 매칭 결과 UI, 온보딩 플로우 |
| 개발 | 구현, 인프라, 데이터, 알고리즘, 테스트 | API 개발, DB 스키마, 파이프라인 구축, 배포 |

**복합 아이템:** 주 파트 1개 + 연관 파트 표기(예: `개발(+기획)`). **주 파트 판정 기준: 최종 산출물을 소유·전달하는 파트.** 3파트 모두 걸치면 산출물 기준으로 분할한 뒤 각각 주 파트를 지정한다. S등급 파트 지정은 대표 최종 확인 — 파트가 브리프 포맷을 결정하기 때문이다.

## 부록 + 카테고리
